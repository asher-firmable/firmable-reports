Background: How SmartLead tracks bounces

When SmartLead sends an email and it fails to deliver, two things can happen:

1 — Silent rejection (85% of bounces)

The receiving mail server rejects the message at SMTP handshake time and returns a 5xx error code directly to SmartLead. SmartLead increments its bounce_count counter, but no email is sent back to the sending mailbox. There is no per-bounce SMTP code stored anywhere in SmartLead's API — just a total count.

2 — NDR email (15% of bounces)

NDR stands for Non-Delivery Report. The receiving mail server accepts the message first, then later determines it can't deliver it, and sends a bounce notification email back to the sending address. This lands in SmartLead's master inbox as an untracked reply and contains the SMTP error code in its body. This is the only source of per-bounce SMTP data available — 147 emails out of 943 total recipient bounces.

Why Healthcare/Education/Insurance have almost no NDR emails: Microsoft 365's DBEB rejection (SMTP code 5.4.1) happens at the SMTP handshake — it's a silent rejection, so no notification email is ever generated. That's also why ~90% of the NDRs we do have are from the minority of bounces where the receiving server sent a notification rather than silently rejecting.

Key Takeaways for the Data Team

  1. Healthcare is the outlier. 21% recipient bounce rate vs 6–11% for other industries. The Healthcare contact list likely contains more Microsoft 365 enterprise tenants with DBEB blocking, or more stale/invalid contacts.
  2. ~90% of hard bounces are 5.4.1 (Microsoft DBEB). The email address is valid, but the Microsoft 365 tenant has a policy blocking all external cold email. This is a deliverability issue, not a data quality issue. The contacts are real people at real companies; they just can't be reached via cold SMTP.
  3. ~9% of hard bounces are genuine bad data — addresses that don't exist (5.1.1), unknown addresses (5.1.0), or deactivated mailboxes (5.2.1). Extrapolated across 943 total recipient bounces, that's roughly ~85 invalid addresses across all US campaigns.
  4. Zero soft bounces detected. No mailbox-full or temporary server issues in the NDR sample. The bounces are permanent, not retriable.
  5. Sender bounces (470 total) are a separate infrastructure problem. The sending accounts are getting flagged. This is unrelated to contact data quality and should be addressed at the sending account / domain reputation level.

Limitations

  • SMTP codes are only available for 147 of 943 recipient bounces (15.6%). The remaining 85% were silently rejected at SMTP level with no notification email generated. SmartLead tracks bounce counts per campaign but does not store per-bounce SMTP reason in any accessible API endpoint.
  • Industry-level SMTP breakdown is only reliable for Cybersecurity (70 NDRs, 57% coverage) and Construction (18 NDRs, 13% coverage). Healthcare, Education, and Insurance have near-zero NDR coverage because Microsoft DBEB bounces are silent rejections.
  • If full SMTP coverage is needed, check whether SmartLead's dashboard has a per-campaign "bounced leads" export that includes bounce reason, or contact SmartLead support for a raw bounce event log.
